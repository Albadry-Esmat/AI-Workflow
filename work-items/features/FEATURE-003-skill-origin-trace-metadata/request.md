# FEATURE-003 — Request: Skill Origin Trace + Approval Tier Metadata

**Origin:** Derived from analysis of the "Autonomous Skill Discovery, Validation, and Generation" feature proposal (2026-06-23). The audit trail / provenance component was adopted as a governance-layer requirement.

---

## Problem Statement

Every skill in `skills/registry.json` currently has a `status` field (`draft`, `active`, `deprecated`, `retired`) but no record of:

- **How** the skill was created — human-authored in a design session, generated by the gap-to-skill pipeline, or migrated from a legacy format
- **Who** (or which session) approved registration, and under what governance tier
- **Whether** a deduplication override was granted at creation time (required by FEATURE-002)

Without this metadata:
- Auditors cannot trace a skill's provenance after the fact
- The governance layer cannot enforce different review requirements for human-authored vs. gap-triggered skills
- Deduplication overrides are silent — there is no persistent record of why two similar skills coexist
- The adaptive pipeline (SKL-050/051, Phase 6 backlog) cannot distinguish a carefully-designed skill from a rapidly generated one when making improvement suggestions

## Requested Behaviour

Every skill registered after this feature is implemented MUST include an `origin_metadata` block in its registry entry:

```json
{
  "origin_metadata": {
    "source":                 "human | gap-triggered | migrated | unknown",
    "created_by_session":     "<session_id or null>",
    "approval_tier":          "standard | expedited | legacy",
    "dedup_override":         false,
    "dedup_override_reason":  null,
    "created_at":             "<ISO-8601>"
  }
}
```

**Backward compatibility:** Existing skills (pre-FEATURE-003) are NOT retroactively updated.
They are implicitly treated as `source: legacy, approval_tier: legacy`.
`validate-skills.sh` MUST issue a **warning** (not an error) for pre-v5.1.0 skills that lack the field.

## Scope

- `skills/schema/registry-entry.schema.json` — add `origin_metadata` object (required for new registrations only)
- `skill-authoring/SKILL.md` — populate `origin_metadata` at Step 5 (registration)
- `docs/governance.md` — add §5.1: Skill Approval Tiers

## Out of Scope

- Retroactive migration of existing 58 skill records
- UI display of provenance metadata
- Enforcement gating on `approval_tier` value (enforcement is a separate guard, not this feature)
