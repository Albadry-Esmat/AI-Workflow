# Glossary — System Terminology

**Version:** 1.0.0 | **Last updated:** 2026-06-16

All domain terms used across the AI Workflow system. This is the single source of truth for terminology. When a term is used in any skill or doc file, its definition must be consistent with this glossary.

---

## A

**ADR (Architecture Decision Record)**
A short document capturing a significant architectural decision — its context, the decision itself, and its consequences. Format: Michael Nygard style (Title, Status, Context, Decision, Consequences). Generated by `documentation-generator`.

**Attack Surface**
The set of all entry points through which an attacker can attempt to enter or extract data from a system. Mapped in `security-review` during threat modeling.

---

## B

**Backpropagation (pipeline)**
When a downstream skill emits a `feedback` entry with `type: "backpropagate"`, the orchestrator re-invokes the named upstream skill with augmented input and invalidates all intermediate artifacts. Capped at 3 iterations per pipeline. See [Architecture](architecture.md#feedback-loops).

**Blue-Green Deployment**
A deployment pattern that maintains two identical environments (blue = current, green = new). Traffic switches instantly on success. Enables zero-downtime releases and instant rollback via DNS flip.

**Bounded Context**
A Domain-Driven Design (DDD) concept defining an explicit boundary within which a domain model applies. Used in `architecture-design` to define module boundaries.

---

## C

**Canary Deployment**
A deployment pattern that routes a small percentage of traffic to the new version, monitors for errors, then gradually increases traffic. Used for high-criticality modules.

**CWE (Common Weakness Enumeration)**
A community-developed list of software and hardware weakness types (e.g., CWE-79: Cross-site Scripting). Used in `security-review` to classify vulnerabilities.

**CQRS (Command Query Responsibility Segregation)**
An architectural pattern that separates read operations (queries) from write operations (commands). Recommended by `architecture-design` for systems requiring scalable reads.

---

## D

**DDD (Domain-Driven Design)**
A software design approach focused on modeling software to match the business domain. `architecture-design` applies DDD principles (bounded contexts, aggregate roots, ubiquitous language).

**Dry Run**
An execution mode (`dry_run: true`) in `doc-maintainer` that produces a maintenance report showing what files would be created or updated, without actually writing any files.

---

## E

**Event Sourcing**
An architectural pattern where state changes are stored as a sequence of events rather than the current state. May be recommended by `architecture-design` for audit-heavy domains.

---

## F

**Feedback Entry**
A structured object in a skill's output `feedback` array. Types: `backpropagate` (re-invoke upstream skill), `info` (informational), `warning` (non-blocking concern). Schema defined in each skill's `$defs.feedback_entry`.

**Feature Flag**
A runtime toggle that enables/disables a feature without a code deployment. Types: `release` (feature rollout), `ops` (operational control), `permission` (access control), `experiment` (A/B testing). Governed by `deployment-strategy`.

---

## G

**Gate (HITL Gate)**
A checkpoint in the pipeline where execution pauses for human approval before continuing. Configured in `pipeline_config.gates`. See [Governance](governance.md#hitl-gate-rules-layer-2).

---

## H

**HITL (Human-in-the-Loop)**
A pattern where a human reviews and approves an AI decision before execution continues. Used at critical pipeline checkpoints: after requirement analysis, architecture design, feature planning, security review, and before deployment.

---

## I

**Idempotent**
A property of a skill where running it twice with identical input produces identical output. All skills in this system are idempotent by design.

---

## J

**JSON Schema (draft-07)**
The schema specification used for all skill input and output validation. The `schema-validator` skill validates data against JSON Schema draft-07. See [http://json-schema.org/draft-07/schema#].

---

## L

**LTR / RTL (Left-to-Right / Right-to-Left)**
Text directionality. English is LTR; Arabic is RTL. The system supports both via CSS logical properties and locale-aware rendering. See [Localization](localization.md).

---

## M

**Metrics (Standard Base Fields)**
Every skill output `metrics` object MUST include: `tokens_in`, `tokens_out`, `duration_ms`, `items_produced`, `version`. Domain-specific fields are additive. See [Monitoring](monitoring.md#per-skill-metrics).

---

## O

**Orchestrator**
The meta-skill (`skills/orchestrator/orchestrator.md`) that drives pipeline execution. It resolves the registry, routes artifacts between skills, validates schemas, enforces HITL gates, and handles feedback loops.

**OWASP Top 10**
A standard awareness document for the ten most critical web application security risks (e.g., A01:2021 Broken Access Control). Used in `security-review` for vulnerability classification.

---

## P

**Pipeline Context (Level 2)**
The in-flight state managed by the orchestrator during a single pipeline execution — current skill, artifacts in transit, gate decisions. Destroyed when the pipeline completes. See [Context Engineering](context-engineering.md#context-model).

**Primitive Obsession**
A code anti-pattern where primitive types (int, string) are used instead of domain-specific value objects. Detected by `clean-code-review`.

---

## R

**Registry** (`skills/registry.json`)
The central catalog of all skills with their version, domain, path, input/output fields, dependency graph (`consumes_from` / `produces_for`), and `status`. The single source of discovery for the orchestrator.

**Rolling Deployment**
A deployment pattern that updates instances gradually, one or a few at a time. No full rollback available — revert is per-instance.

---

## S

**Saga**
A distributed transactions pattern that breaks a long-running transaction into a sequence of local transactions with compensating actions for rollback. May be recommended by `architecture-design`.

**Semver (Semantic Versioning)**
The `MAJOR.MINOR.PATCH` versioning scheme used for all skills and documentation files. MAJOR = breaking change, MINOR = new capability, PATCH = fix/clarification. See [Versioning](versioning.md).

**Session Context (Level 3)**
The persisted state across multiple pipeline turns. Managed by the context memory protocol (`skills/memory/context-protocol.md`). Retains last 3 skill outputs; supports resumption after HITL pauses or token exhaustion.

**SOLID Principles**
Five object-oriented design principles: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion. Enforced by `clean-code-review`.

**STRIDE**
A threat modeling framework: Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege. Applied per module in `security-review`.

---

## T

**Token Budget**
The maximum token allocation for a session. Types: Quick (32K), Standard (64K), Deep (128K). Enforced by the context memory protocol. See [Context Engineering](context-engineering.md#token-budget-per-session).

**Transitive Update**
A documentation update triggered by a primary change — e.g., a skill change (`primary`) may also require updating `development-standards.md` (`transitive`) if new standards are introduced. Defined in the `doc-maintainer` Secondary Updates table.

---

## U

**Ubiquitous Language**
A DDD concept — a shared vocabulary between developers and domain experts applied consistently in code and design. Used in `architecture-design` during module boundary definition.
